## TD-1: Reduce size of files and functions

Files should have a clear single responsibility. Functions should do one thing. Current violations:

- `container/ollama-runner/src/index.ts` at 3,361 lines is the worst offender
- `src/index.ts` (929), `src/ipc.ts` (897), `src/channels/telegram.ts` (1,193) are all over the line
- Goal: no source file over ~400 lines, no function over ~50 lines
- Approach: extract cohesive chunks into named modules, not just "utils" dumping grounds

This isn't just aesthetics — large files make Claude's context window less effective when it's working on the codebase, and they make test coverage harder to add.

## TD-2: Implement a code linter with structural, SLA, and performance rules

Currently there is NO linter — only Prettier (formatter). This means:
- Unused variables, unsafe `any` casts, missing return types slip through
- No enforcement of file/function size limits
- No performance anti-patterns caught (e.g. blocking calls in async paths)
- No consistency rules per module/component

Recommendation:
- Add **ESLint** (or **Biome** for speed — it also replaces Prettier) with a strict TypeScript config
- Add **eslint-plugin-sonarjs** or similar for complexity/cognitive load rules (max lines per function, cyclomatic complexity)
- Add custom rules or thresholds per component type (e.g. container code vs. src vs. channels)
- Hook into the existing Husky pre-commit pipeline
- Add `lint` to CI / build step so it fails fast

The linter should enforce minimums per module — not just "nice to have" but a hard gate. It all adds up.
